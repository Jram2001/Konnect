01system overview
what it does: user tells us what entities to watch (stocks, people, companies) every morning a cron job runs: fetch news → filter relevant → AI extracts entities + events + scores → match to watchlists → send email what it builds over time: a knowledge graph of entities and their connections to real-world events every AI extraction is a node in the graph queryable: "what happened to tesla this week" = one indexed query
Email-only MVP. No frontend dashboard. No auth. No real-time. Ship fast, validate, add UI later.

news.service
Fetches articles from NewsAPI. Hashes URLs for dedup. Stores raw articles. Returns unprocessed batch.
ai.service
Sends articles to Claude. Gets back entity_key, display_name, events, scores. Generates mood summaries. Parses and validates JSON output.
entity.service
Upserts entities by entity_key. Manages macro_group_keys assignment. Canonical registry for all known entities.
digest.service
Orchestrates per-user digest. Queries micro + macro connections. Merges, ranks, deduplicates. Creates digest documents.
email.service
Nodemailer SMTP transport. Renders HTML email template from digest data. Sends and tracks delivery status.

04daily pipeline
06:00 AM — cron fires
step 1 — ingest
  fetch news from API
  hash URLs, skip duplicates
  store new articles (is_processed = false)

step 2 — pre-filter (free, no AI)
  collect all entity display_names from all user watchlists
  string match against article title + body
  only articles mentioning a watched entity pass through
  ~80% of articles skipped = ~80% cost saved

step 3 — process (Claude API)
  for each relevant article:
    send to Claude → get [{ entity_key, display_name, event_text, event_category, impact_score, reasoning, is_macro }]
    for each extraction: upsert entity by entity_key
    bulk insert connections
    mark article as processed

step 4 — compose
  for each active user:
    collect entity_keys from watchlist
    lookup macro_group_keys from entities
    query connections: entity_key $in watchlist (micro)
    query connections: macro_group_keys $in groups, is_macro=true (macro)
    merge + dedup + sort by abs(impact_score)
    Claude generates mood summary + subject line
    create digest doc

step 5 — deliver
  for each pending digest:
    render email HTML from digest items
    send via nodemailer
    mark status = delivered
05entity resolution
problem: articles say "Tesla", "TSLA", "Tesla Inc", "Tesla Motors" all the same entity. must resolve to one record. also: "Tesla" the company vs "Tesla" the scientist different entities. same word. must disambiguate. solution: Claude returns an entity_key in format: name_type "Tesla Inc" → tesla_company "TSLA" → tesla_company "Tesla Motors" → tesla_company "Nikola Tesla" → nikola_tesla_person how it works: 1. AI prompt enforces: "return entity_key as lowercase name_type" 2. naming convention is self-normalizing — no registry list needed 3. entity.service upserts by entity_key (findOneAndUpdate, upsert: true) 4. all downstream references use entity_key, never display_name 5. display_name is for humans only (emails, UI)
06macro matching
what: events that affect an entire sector, not just one entity. "Fed raises rates" → affects all banking stocks. user watches "jpmorgan_company" → gets the Fed event via macro match. how (MVP — manual only): 1. admin seeds macro_groups: "ev_sector", "banking", "ai_industry" 2. admin assigns group_keys to entities: tesla_company → ["ev_sector", "big_tech"] jpmorgan_company → ["banking"] 3. when Claude flags is_macro = true on an event, it also returns which macro_group_keys are affected 4. digest builder collects user's group_keys from their watchlist entities 5. queries connections where macro_group_keys overlap + is_macro = true later (hybrid): pass group list in Claude prompt. AI picks from approved list. suggestions queue for new groups.
